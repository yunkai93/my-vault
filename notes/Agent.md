# Agent

> [!abstract]
> Agent 是以 LLM 为决策核心、可调用工具并根据反馈持续完成多步任务的系统。

Agent：以 LLM 为决策核心，能够调用工具、读取环境反馈、围绕目标持续执行多步任务的系统。
它的本质不是“更会聊天”，而是进入一个闭环：理解目标 -> 决定动作 -> 执行动作 -> 读取反馈 -> 继续修正。

## 先理解 Agent 是什么
### Agent 和工作流的区别
- 工作流（Workflow）：路径由代码预先写死，分支和顺序基本确定
- Agent：路径由模型在运行时动态决定，可以根据反馈临时调整策略

可以用一句话区分：
- 工作流更像“按图施工”
- Agent 更像“边做边判断”

### Agent 的最小闭环
一个最小可用 Agent 通常包含：
- 目标：要完成什么
- 上下文：已知信息、约束、历史记录
- 推理：模型如何分析问题和决定下一步
- 行动：调用工具、访问系统、执行操作
- 观察：读取工具结果或环境反馈
- 记忆：保存重要状态，支持多轮和长任务

可以抽象成一个循环：
1. 理解当前目标和状态
2. 决定下一步动作
3. 调用工具或执行操作
4. 获取反馈
5. 判断是否结束，否则继续下一轮

### Agent 的核心价值
- 把模型从内容生成器变成任务执行器
- 用工具和环境反馈减少纯语言生成带来的幻觉
- 让复杂任务可以拆成多步逐步完成
- 让系统具备一定的自修正能力

## Agent 由什么组成
### 模型
模型负责理解任务、规划步骤、选择工具、综合结果。

### 工具
工具是 Agent 连接外部世界的手和脚，例如：
- 搜索
- 数据库查询
- 代码执行
- 文件读写
- 调用业务 API

工具设计通常比工具数量更重要。好工具应该：
- 输入输出清晰
- 错误可见且可修复
- 尽量面向任务，而不是暴露过底层的原始 API

### 上下文
上下文决定模型每一步能看到什么，常见包括：
- 系统规则
- 当前任务说明
- 外部知识
- 运行时状态
- 历史轨迹
- 记忆

### 记忆
常见可分为：
- 工作记忆：当前任务过程中的临时状态
- 程序性记忆：做事的方法和规则
- 情景记忆：过去某次尝试、失败或成功经验
- 语义记忆：长期保存的事实性知识

### 反馈机制
反馈机制决定 Agent 是否真正闭环，常见反馈来源包括：
- 搜索结果
- API 返回值
- 编译/运行/测试结果
- 用户确认
- 评分器或判别器输出

## 什么任务适合 Agent
更适合 Agent 的任务通常满足：
- 目标相对明确
- 可以拆成多步
- 外部反馈可获得
- 成功与失败至少部分可验证

## 主流设计模式
先记住一个总判断：
- ReAct 是基础范式
- Plan and Execute / ReWOO / LLM Compiler 主要解决速度和成本问题
- Reflection / Reflexion / LATS 主要解决质量、稳定性和探索深度问题

### 1. ReAct
ReAct = Reasoning + Acting。
核心循环是 Thought -> Action -> Observation。

基本过程：
1. Thought：分析现状，决定下一步
2. Action：调用一个工具或执行一个动作
3. Observation：读取结果
4. 回到下一轮 Thought，直到完成任务

它和 Chain-of-Thought 的关键区别是：
- CoT 主要在模型内部推理
- ReAct 把推理和外部行动连接起来

优点：
- 灵活，能根据实时反馈动态调整
- 可解释性强，便于调试
- 对检索问答、代码执行、网页操作、多步任务很有效
- 能显著降低纯靠参数记忆带来的幻觉

缺点：
- 每一步几乎都要调用模型，速度慢、成本高
- 容易只盯着眼前一步，缺少全局规划
- 工具调用过多时，整体延迟明显

适用场景：
- 需要边做边看的任务
- 工具结果会显著影响后续动作的任务
- 编程、调试、检索问答、网页操作

一句话理解：ReAct 是 Agent 最核心的原型，后面很多模式都是在补它“慢”和“局部视角强”的短板。

### 2. Plan and Execute
Plan and Execute 把任务拆成两个阶段：
- Plan：先生成一个较完整的计划
- Execute：再按计划依次执行

它相对 ReAct 的变化是：
- 把频繁决策改成先整体规划
- 让昂贵模型少参与，执行阶段可交给更轻的组件

优点：
- 降低大模型调用次数
- 成本更低，执行更稳定
- 更容易做全局规划，减少局部最优

缺点：
- 计划通常偏静态
- 中途遇到变化时，适应性弱
- 某一步失败后，往往需要重新规划

适用场景：
- 流程较明确但步骤较多的任务
- 报告生成、数据处理、固定业务流程

一句话理解：它更像“带规划能力的工作流”，灵活性不如 ReAct，但更省。

### 3. ReWOO
ReWOO = Reasoning Without Observation。
它会一次性生成一串推理和工具调用计划，并使用变量引用前面步骤的结果，例如 `#E1`、`#E2`。

它相比 Plan and Execute 的改进是：
- 不只是列步骤，还显式表达数据依赖
- 通过变量传递结果，减少中间反复调用模型

优点：
- Token 使用效率更高
- 适合信息链式传递任务
- 比简单计划列表更适合处理多步依赖

缺点：
- 缺少持续观察后的动态重规划
- 本质仍偏串行
- 某一步出错时，整体鲁棒性一般

适用场景：
- 信息抽取
- 链式检索
- 结构化多步查询

一句话理解：它是在牺牲部分灵活性，换取更高的执行效率。

### 4. LLM Compiler
LLM Compiler 的核心思想是：
- 让模型先生成任务依赖图，而不只是线性步骤
- 再由调度器按依赖关系并行执行任务

它本质是在 Agent 设计里引入 DAG 调度。

优点：
- 能并行执行独立工具调用
- 执行速度显著快于纯串行模式
- 结构清晰，便于理解任务依赖

缺点：
- 工程复杂度高
- 仍然依赖前置规划正确
- 对失败恢复和动态改道支持有限

适用场景：
- 可并行的信息收集
- 多源数据整合
- 多个独立子任务可同时推进的任务

一句话理解：它解决的核心不是“更聪明”，而是“更快”。

### 5. Basic Reflection
Basic Reflection 是在任务输出后增加一个显式反思步骤，让模型自己检查自己的结果。

常见形式：
1. 先生成初稿或初始方案
2. 再让模型批判它的问题
3. 根据批评结果重新生成

优点：
- 实现简单
- 对写作、总结、代码草稿等任务很有效
- 能改善一致性、完整性和表达质量

缺点：
- 如果没有外部反馈，可能只是自说自话
- 不能保证真的发现关键错误
- 在客观验证强的任务上，往往不如真实工具反馈可靠

- 写作

一句话理解：当你拿不到真实世界反馈时，可以先让模型自己挑刺。

### 6. Reflexion
Reflexion 是更强版本的反思框架。
它不仅在单次任务中反思，还会把失败经验总结成文本记忆，在下一次尝试中继续使用。

核心过程：
1. 执行任务
2. 评估成功或失败
3. 生成反思文本
4. 把反思写入记忆
5. 下一轮尝试读取这些经验，避免重复犯错

优点：
- 能从失败中积累经验
- 不需要微调模型，也能提升多步任务成功率
- 对代码、顺序决策、试错型任务效果明显

缺点：
- 依赖评估和反思质量
- 记忆管理本身会变成新问题
- 任务过长时，记忆容量和检索方式会成为瓶颈

适用场景：
- 需要多次试错的问题
- 编程、调试、复杂交互式任务

一句话理解：它不是只会复盘，而是会把复盘结果记住，下次少犯同样的错。

### 7. LATS
LATS = Language Agent Tree Search。
它把语言模型和树搜索结合起来，在每个关键节点探索多条可能路径，而不是只走一条。

你可以把它理解为：
- ReAct 只走一条路
- LATS 会同时尝试多条路，并评估哪条更值得继续扩展

优点：
- 更擅长探索复杂决策空间
- 不容易过早陷入单一路径
- 结合了行动、反思和搜索的优势

缺点：
- 成本非常高
- 实现复杂
- 往往需要环境支持回溯或状态复原

适用场景：
- 高探索性、高分支的复杂任务
- 复杂规划、多步编程、策略搜索

一句话理解：它适合少数高价值、强探索任务，不适合作为默认架构。

## 怎么理解这些模式的关系
可以把它们看成三条演化线：

### 1. 反馈闭环线
- ReAct 是起点
- 强调边行动边观察
- 核心价值是让模型和真实环境形成闭环

### 2. 性能优化线
- Plan and Execute
- ReWOO
- LLM Compiler

这条线主要解决 ReAct 太慢、太贵的问题。
方法包括：
- 提前规划
- 减少中间模型调用
- 并行执行工具

### 3. 自我修正与探索线
- Basic Reflection
- Reflexion
- LATS

这条线主要解决两个问题：
- 单次输出不够稳
- 单一路径探索不够深

方法包括：
- 自我批判
- 记住失败经验
- 同时探索多条路径

## 怎么选设计模式
### 模式对比表
| 模式               | 核心思想           | 最大优势             | 主要代价          | 更适合的任务          |
| ---------------- | -------------- | ---------------- | ------------- | --------------- |
| ReAct            | 边推理边行动边观察      | 灵活，反馈闭环强         | 慢，成本高         | 编程、调试、检索问答、网页操作 |
| Plan and Execute | 先规划再执行         | 更省调用，更容易全局规划     | 中途变化时适应性弱     | 固定流程、多步骤执行      |
| ReWOO            | 预先规划并用变量传递中间结果 | Token 效率高，依赖表达清晰 | 缺少动态重规划       | 链式检索、结构化查询      |
| LLM Compiler     | 先生成依赖图，再并行调度   | 速度快，可并行          | 工程复杂，恢复难      | 多源收集、并行子任务      |
| Basic Reflection | 先产出，再自我批改      | 简单，能提升文本质量       | 没有外部反馈时不一定可靠  | 写作、总结、代码润色      |
| Reflexion        | 反思后把经验写入记忆     | 能从失败中学习          | 依赖评估质量，记忆管理复杂 | 多次试错、复杂交互式任务    |
| LATS             | 同时探索多条路径并评估    | 探索能力强，不易早早走死     | 成本极高，实现复杂     | 高价值、高分支决策任务     |

### 一个简洁的选型指南
- 任务需要频繁读取真实反馈：优先 ReAct
- 任务流程稳定但步骤很多：优先 Plan and Execute
- 步骤之间数据依赖明显：优先 ReWOO
- 子任务可以并行：优先 LLM Compiler
- 缺少客观反馈，但需要改稿和自检：优先 Basic Reflection
- 需要多次试错，并希望从失败中积累经验：优先 Reflexion
- 决策空间很大，必须探索多条路线：优先 LATS

> [!tip]
> 一个实用经验：默认先从 ReAct 或“ReAct + 少量工程增强”开始，只有在明显遇到速度、成本、质量或探索深度瓶颈时，再引入其他模式。

## 工程上真正重要的点
### 1. 先判断任务是否真的需要 Agent
如果固定流程就能解决，工作流通常更简单、更稳、更便宜。

### 2. 优先设计反馈，不要先堆复杂推理
反馈越真实，Agent 越不容易幻觉。
编程场景里的编译、测试、运行结果，就是最好的监督信号。

### 3. 工具设计优先于提示词堆叠
稳定的工具接口、明确的错误信息、可恢复的执行结果，往往比长 prompt 更重要。

### 4. 让状态外化
长任务不要只靠上下文硬撑。
把计划、进度、关键中间结果写到外部状态里，系统才更可恢复。

### 5. 不要默认上多 Agent
多 Agent 会引入通信、协调、上下文隔离和错误传播问题。
通常先把单 Agent 做好，再考虑拆分。

### 6. 评测要看环境结果，不只看文本输出
对于 Agent，真正重要的是任务是否完成，而不是回答看起来是否像完成了。

## Claude Code 这类系统怎么理解
像 Claude Code 这类编程 Agent，可以把它理解为一种偏 ReAct 的系统：
- 模型负责理解目标和决定下一步
- 通过读文件、搜索、执行命令、编辑代码等工具与环境交互
- 根据命令输出、测试结果、文件内容继续调整

它之所以比“单次问答”强很多，不是因为它只会写代码，而是因为它能持续获取外部反馈并据此修正行为。

同时，它又会叠加一些工程化能力，例如：
- 任务拆分
- 外部状态管理
- 规则和记忆加载
- 工具权限控制
- 必要时并行处理子任务

所以它通常不是某一篇论文的直接实现，而是以 ReAct 为骨架，再融合工作流、记忆、规则、反思与工程控制的混合系统。

## 一句话总结
Agent 不是一个神秘的新物种，它本质上就是让 LLM 围绕目标进入闭环：会思考、会调用工具、会读取反馈、会继续修正。

主流设计模式的差别，本质上都在回答这几个问题：
- 要不要实时反馈
- 要不要提前规划
- 要不要自我反思
- 要不要并行执行
- 要不要探索多条路径
