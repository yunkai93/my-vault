# AI应用技术

> [!abstract]
> 这是一篇 AI 应用方法总览笔记，用来串起模型、提示、检索、工具调用、MCP 与 Agent。
## LLM
Large Language Model 大语言模型。
Decoder-Only 是只保留 Transformer 中 Decoder 部分的模型架构。
它的核心方式是根据前文，按顺序逐个预测下一个 token。
因此它本质上是一种自回归生成模型，天然适合续写、问答、对话和代码生成。
它的关键机制是 masked self-attention：
- 可以看前面的 token
- 不能看后面的 token
- 从而保证生成过程符合时间顺序

## Prompt Engineering
提示词（Prompt）是用来引导模型按照特定意图生成输出的输入指令。
Prompt Engineering 是围绕大模型设计输入的方法，目标是让输出更准确、更稳定、更可控。

一个有效的 prompt 通常包含这些要素：
- 任务：要模型做什么
- 上下文：模型完成任务所需背景
- 约束：范围、语气、长度、边界条件
- 输出格式：列表、表格、JSON、固定模板
- 示例：给出目标输入输出样例

可概括为：任务 + 上下文 + 约束 + 输出格式 + 示例。

核心原则：
- 先把任务说清楚，再谈技巧
- prompt 要具体，避免模糊表述
- 复杂任务尽量拆成步骤
- 需要稳定格式时，明确指定输出结构
- 需要更稳的结果时，可以提供少量示例
- 效果不好时，先补上下文和约束

使用建议：
- 简单任务：直接说清目标即可
- 复杂任务：补上下文、拆步骤、定格式
- 程序消费结果：优先要求结构化输出
- 事实敏感任务：优先结合 RAG 或外部资料

## RAG
RAG 是 Retrieval-Augmented Generation，即检索增强生成。
它的核心思路是先从外部知识库中检索相关内容，再把检索结果作为上下文交给模型生成答案。

RAG 把知识分成两部分：
- 参数记忆：模型本身学到的通用语言能力和压缩知识
- 外部记忆：可检索、可更新的文档或知识库

一个典型的 RAG 流程包括：
- 将文档切分并建立索引
- 根据问题检索相关片段
- 把检索结果与问题一起交给模型生成回答

RAG 的价值让回答建立在外部资料之上，从而提高可更新性、可验证性和事实准确性。
因此它常用于知识库问答、企业搜索、客服助手、文档问答和个人笔记检索等场景。

## Function Calling
Function Calling 是让大模型根据用户请求，生成对某个函数或工具的结构化调用参数，再由外部程序真正执行。
它的作用是把语言理解和系统动作连接起来，让模型能够借助外部能力完成查询、检索、调用接口和执行业务流程。

一个典型流程包括：
- 向模型声明可用函数及参数结构
- 模型判断是否需要调用函数
- 模型输出函数名和参数
- 外部程序执行函数
- 将结果返回给模型生成最终回答

Function Calling 的关键点是：模型负责决定调用什么和生成参数，但不负责真正执行。
因此函数描述要清楚，参数仍要校验，工具结果也不能无条件信任。
它是当前 LLM 应用中连接模型与外部系统的常见模式，也是 Agent、RAG、工作流自动化的重要基础能力。

## MCP
MCP 是 Model Context Protocol，即模型上下文协议。
它是一种开放标准，用来规范 AI 应用如何以统一方式连接外部工具、数据源和服务。
相比每个 AI 平台各自集成一套工具，MCP 试图用统一协议减少接入碎片化、降低平台绑定，并提升安全性和复用性。

一个基本结构通常包括：
- Host：承载 AI 的应用
- MCP Client：负责协议通信的客户端层
- MCP Server：对外暴露工具、资源或服务
- 外部资源：数据库、文件系统、API、系统工具等

## Agent
Agent 是以 LLM 为决策核心、能调用工具并根据环境反馈持续完成多步任务的系统。
它的关键不是“更会生成内容”，而是形成了“目标 - 动作 - 反馈 - 修正”的闭环。
更完整的整理见：[[Agent]]
- 基础范式：ReAct
- 性能优化：Plan and Execute、ReWOO、LLM Compiler
- 自我修正与探索：Basic Reflection、Reflexion、LATS
- 工程重点：反馈设计、工具设计、状态外化、评测方式


## Multi-Agent
由多个分工协作的 Agent 共同完成任务，通过拆分任务与隔离上下文解决单 Agent 系统难以处理的复杂问题。 需要谨慎使用以避免 Token 消耗量大、协作效率低、系统复杂度过高等问题。

## Context Engineering
Agent 运行中需要提供给 LLM 的一切相关信息（如对话历史、用户输入、背景知识、工具结果等）都是上下文。 上下文工程关注如何高质量筛选、压缩和组织上下文，从而最大化模型决策与推理能力。

## Agent Skill
Agent Skills 是一种轻量级的开放格式，用于将一整套 Agent 能力（prompt、工具脚本、知识文件等） 封装为可复用模块，从而实现低门槛分享与复用。 Agent Skill 本质上约等于一个子 Agent。 Agent Skill 特别适合 SOP 的沉淀和复用（离职的同事终将化作温暖的 Skill）。 Agent 会在运行过程中按需激活不同 Skills、按需读取和使用 Skills 文件包里的内容（渐进式披露）。
