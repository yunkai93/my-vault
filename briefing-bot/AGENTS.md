# Briefing Bot 方案说明

这个目录用于实现 `my-vault` 的 AI 早报自动化。

当前先采用一版**简化方案**，目标是先稳定跑通，再逐步增强。

## 已确认目标

每天自动执行一次：

1. 抓取主流 AI / 设计资讯媒体当前可获取的**最新一期**
2. 对内容做去重、归类、简化整理
3. 生成符合 `my-vault` 规则的早报内容
4. 写入 `news/AI早报.md`
5. 自动 `git commit` 并 `push origin main`
6. 再将同样内容推送到飞书群机器人

## 已确认运行方式

- 代码位置：`my-vault/briefing-bot/`
- Git 分支：`main`
- 调度方式：`systemd timer`
- 执行时间：每天北京时间 `08:50`

## 简化方案原则

第一版**不做全天分时采集**。

而是采用：

- 每天 `08:50` 固定执行一次
- 逐个抓取各媒体此刻能拿到的**最新一期**
- 不强求一定是“当天刚发布”
- 抓不到某个源就跳过
- 只要整体内容足够，就生成并发布

也就是说，第一版是“**固定时刻抓各源最新产物**”，不是“全天盯着所有源等它们出刊”。

这样做的原因：

- 实现简单
- 运维简单
- 更容易排错
- 更适合先落地 MVP

## 内容时间窗口

第一版建议加一个轻量时间限制：

- 默认只接受**最近 72 小时**内的最新内容
- 超过 72 小时的源，默认跳过

这样可以避免某个媒体长时间没更新时，把过旧内容混进早报。

## 推荐主流媒体池

当前建议把以下媒体作为第一版主要输入源。

### AI 主源

1. `The AI Rundown`
   - 定位：AI 总览型日更早报
   - 特点：`Top 5 stories + So What`
   - 适合：抓当天最值得关注的 AI 新闻主线

2. `TLDR AI`
   - 定位：主流 AI 快讯
   - 特点：栏目清晰，归档公开
   - 适合：模型、Agent、研究、产品动态

3. `The Neuron`
   - 定位：AI 趋势、Agent 热点、工具圈热点
   - 特点：常覆盖热门 Agent、工作流、圈内舆情
   - 注意：抓取可能受 Cloudflare 影响

4. `Superhuman AI`
   - 定位：AI 工具、工作流、实操导向 newsletter
   - 适合：工具热点、使用场景、设计师协作流

5. `Futurepedia Newsletter`
   - 定位：AI 工具、工作流、创意工具导向
   - 适合：AI 工具更新、设计相关 AI 工具动态

### 设计补充源

6. `TLDR Design`
   - 定位：设计领域主流快讯
   - 适合：设计工具、设计行业动态、AI 设计相关新闻

7. `AI in Design`
   - 定位：AI + 设计窄域精选
   - 适合：设计师与 AI 协作、UX、brand、creative tools

8. `优设网 / UISDC`
   - 链接：`https://www.uisdc.com/`
   - 定位：中文设计领域资讯与教程站
   - 适合：中文设计圈热点、设计工具、AI 设计相关文章补充
   - 用途：作为设计领域中文补充源，优先用于补充国内设计圈动态，不作为唯一主源

### 周度校准源

9. `The Batch`
   - 定位：周更 AI 新闻与分析
   - 第一版不作为日报主源
   - 后续可用于周度校准，避免日报内容过度偏工具化

## 第一版早报模板方向

最终不直接照搬某一家 newsletter，而是采用自有极简模板。

建议结构：

```md
## YYYY-MM-DD

_更新时间：YYYY-MM-DD HH:mm_

### 今日判断
一句话概括今天最值得注意的变化。

### Agent Watch
- 条目

### Model Watch
- 条目

### Design x AI
- 条目

### 快速雷达
- 3-6 条一句话简讯

### 值得跟进
- 1-3 条后续观察点

### 来源参考
- 列出本次实际采用的媒体源
```

## 内容写入规则

生成结果必须遵守仓库既有规则：

- 只写入 `news/AI早报.md`
- 该文件是单一滚动文件
- 按日期倒序追加
- 同一天如果重跑，应该**覆盖当天块**，而不是重复新增

## 发布规则

默认流程：

1. 抓取并整理内容
2. 生成早报 Markdown
3. 更新 `news/AI早报.md`
4. `git pull --rebase`
5. `git commit`
6. `git push origin main`
7. push 成功后再推送飞书

## 第一版容错原则

- 某个媒体抓不到：跳过
- 某个媒体内容过旧：跳过
- 只要总体内容足够：照常生成
- 如果整体有效内容太少：
  - 可以生成降级版
  - 或直接记日志并跳过发布
  - 最终策略待定

## 当前未定项

以下内容还没有最终确认：

1. 飞书机器人 webhook / secret
2. 失败时是否要单独发异常通知
3. “有效内容太少”的阈值
4. 各媒体的具体抓取方式
   - RSS
   - 公开归档页解析
   - 浏览器抓取
5. 去重规则细节
6. 飞书消息格式

## 当前结论

这套系统的第一版方向已经明确：

- 不做复杂全天采集
- 不追求严格“当天每家都出刊后再抓”
- 只在每天 `08:50` 抓各源当前最新一期
- 抓到什么整理什么
- 以稳定、简洁、易维护为第一优先级
